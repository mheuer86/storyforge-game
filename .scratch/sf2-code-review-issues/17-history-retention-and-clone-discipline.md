# Decide SF2 history retention and clone discipline

Status: decided
Category: enhancement
**Type:** HITL
**Source:** sf2 code review pass 1, finding #2

## What to build

SF2 state is treated as both mutable and immutable. The turn pipeline, apply-patch, prune-graph, and persistence normalizer each clone at different boundaries, while some helpers mutate in place. At the same time, `history.turns` is unbounded and stores `narratorAnnotationRaw` indefinitely.

This issue records the architecture decision before implementation work:

**Option A: Mutable drafts with named clone boundaries.**
- Clone once at route or pipeline entry
- Mutate draft state inside runtime helpers
- Document naming convention: `applyX` mutates, `withX` returns new state

**Option B: Immer-style immutable reducer.**
- One canonical producer boundary
- Draft mutation inside reducers only
- Remove ad hoc `structuredClone` calls

Also decide retention:
- Keep full `history.turns` forever in canonical state
- Cap raw per-turn payloads and keep compact `recentTurns`
- Move replay-heavy raw annotations to an export/debug artifact outside canonical state

## Decision

Chosen mutation discipline: **Option A — mutable drafts with named clone boundaries**.

Conventions:
- Route/pipeline entry points clone before mutation.
- `applyX` helpers may mutate their draft argument.
- `withX` helpers must return a new value without mutating inputs.
- Broad reducer rewrites are out of scope for this ticket bundle.

Chosen retention policy: **keep canonical turn summaries, cap raw replay-heavy payloads**.

Policy:
- `history.turns` remains the canonical chronological record for save compatibility.
- Compact `history.recentTurns` remains the bounded prompt/retrieval surface.
- `history.turns[].narratorAnnotationRaw` should be capped or externalized in a follow-up so old saves still load and replay fixtures can opt into raw annotations where needed.
- Debug/export artifacts may preserve full raw annotations outside canonical state.

Save compatibility risk: removing or capping raw annotations changes what older debug/replay exports can reconstruct. The follow-up must be additive or migration-safe and must preserve existing saved campaigns.

Replay expectations: fixtures that assert raw annotation behavior should either keep raw data inside their fixture input or assert the compact retained fields after normalization; capped/externalized raw fields must not silently break `npm run sf2:replay`.
