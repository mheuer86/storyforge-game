# Decide firewall fate (enforce or demote)

Status: implemented
Category: enhancement
**Type:** HITL
**Source:** sf2 code review pass 1, finding #1

## What to build

`firewall/actor.ts:9` hardcodes `currentMode = 'observe'`. Comment says enforcement flips after one Ch1 playthrough of zero legitimate rejections — but no auto-flip mechanism, no counter, no decision criterion in code. Every `observeActorFirewallWrite` call pays the full cost (evaluate, ring buffer, conditional console.warn, return event) but downstream nothing reads `decision.allowed`. Cost without enforcement.

Two viable shapes; pick one in this issue, then implement.

**Option A: Commit to enforce.**
- Add a counter that auto-flips when N (e.g. 100) consecutive observed writes produce zero `reason !== undefined`
- The flip is one constant + telemetry signal
- Illegal writes get rejected at the route layer in enforce mode

**Option B: Demote to dev-only assertion.**
- Behind `process.env.NODE_ENV !== 'production'`, throw on illegal writes
- Remove the ring buffer, the memory sink, the per-write console.warn
- The contract still lives in the type definitions

The current state (full instrumentation cost without enforcement payoff and without a path to enforcement) is the worst trade.

## Decision

Chosen: **Option B — demote to dev-only assertion**.

Reasoning: the firewall is a developer contract, not a player-facing runtime gate. Keeping observe-mode instrumentation in production adds per-write cost without an enforcement path. Dev/test should fail loudly when role ownership drifts; production should avoid the ring buffer and warning noise.
