## Why this ticket exists

Investigation, combat, and exploration now have SF2 procedure/packet code, but the current implementation is uneven across three layers:

- authoritative state and activation
- Narrator packet/beat-mode consumption
- player-facing UI representation

These mechanics must be wired as real gameplay systems, not just fixture-backed packet helpers. This ticket captures the current state so the next shaping pass can decide the source-of-truth contract and split implementation work cleanly.

## Current state snapshot

### Investigation

What exists:

- `lib/sf2/procedure-investigation.ts` builds an investigation synthesis packet with known facts, hypotheses, contradictions, open questions, and next leads.
- It activates when an active investigation procedure exists or when there are at least two non-consumed clues.
- `buildMechanicsPacket` adds a lightweight investigation active module when attached or floating clues exist.
- `renderScenePacket` renders both the light active module and the richer `Investigation synthesis` specialized packet.
- Focused replay fixture passes: `fixtures/sf2/replay/procedure-investigation-synthesis.json`.

Likely gap:

- There is no `investigation` beat mode. Investigation-like input currently maps into `exploration` beat mode via search/inspect/investigate patterns.
- UI only partially represents investigation through `Intel / Case Board`: threads, attached clues, and floating clues. It does not show the synthesis packet's hypotheses, contradictions, open questions, known facts, or next leads.

### Exploration

What exists:

- `lib/sf2/procedure-access-exploration.ts` derives exploration runtime from active `campaign.procedures` records: nodes, routes, hazards, current node, and ambient alertness.
- `buildMechanicsPacket` collects active access/exploration packets and `renderScenePacket` renders them under `Specialized procedure packets`.
- Focused replay fixtures pass:
  - `fixtures/sf2/replay/procedure-access-exploration-runtime.json`
  - `fixtures/sf2/replay/procedure-access-exploration-dedup-boundaries.json`

Likely gap:

- `deriveSf2BeatMode` selects exploration if `state.world.exploration` exists, or from player intent, but the specialized exploration packet is derived from active `campaign.procedures`. Those sources can diverge.
- `buildMechanicsPacket` only adds the light `activeModules` exploration entry from `state.world.exploration`, not from an active exploration procedure.
- UI `LocationsPanel` displays current location and area nodes from location/world state, but not specialized exploration procedure data such as routes, route statuses, hazards, ambient alertness, or procedure-scoped resources.

### Combat

What exists:

- `lib/sf2/procedure-combat.ts` builds a combat packet with round, initiative, active enemies, HP, AC, status, player position, hazards, objectives, exit conditions, and parent procedure ids.
- Combat beat mode takes priority when `state.world.combat.active` is true.
- `buildMechanicsPacket` attaches combat runtime data when `state.world.combat.active` is true.
- Focused replay fixture passes: `fixtures/sf2/replay/procedure-combat-packet-coexists-operation.json`.

Likely gap:

- Combat packet activation depends on `state.world.combat.active`, not just an active `campaign.procedures` combat record.
- I did not find a live SF2 writer that sets `world.combat` in normal play.
- Narrator mechanical effects currently allow `hp_delta`, `credits_delta`, `inventory_use`, `set_location`, `scene_end`, and `set_scene_snapshot`; they do not expose combat start/update/end markers.
- UI does not appear to have an SF2 combat HUD showing enemies, round, status, initiative, objectives, or exit conditions. V1 has combat UI; SF2 does not appear to.

## Cross-cutting source-of-truth problem

There are two state shapes in play:

- Newer procedure kernel: `campaign.procedures` plus specialized packet builders.
- Older/lightweight world flags: `world.combat`, `world.exploration`, and `world.operation`.

Current code often derives rich packets from `campaign.procedures`, but beat mode and light active modules sometimes depend on `world.*` flags. Shaping needs to decide which layer is canonical for each mechanic, and whether legacy world fields become:

- compatibility projections from procedure runtime
- short-term activation flags maintained alongside procedures
- deprecated fields removed from the live path

## Candidate UI surfaces to shape

Investigation:

- Case board section for active question, known facts, live hypotheses, contradictions, and next leads.
- Clues remain visible, but are organized under the active question rather than only under threads.

Exploration:

- Location panel extension for current node, known routes/exits, blocked or risky routes, active hazards, ambient alertness, and useful route resources.
- Should coexist with access/infiltration status when both are active.

Combat:

- Compact combat HUD with round, enemy HP/status, player position, objectives, hazards, and exit condition hints.
- Should coexist with operation/access UI when combat erupts inside a procedure.
