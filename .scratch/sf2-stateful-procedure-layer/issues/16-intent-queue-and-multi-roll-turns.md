## What to build

Support ordered player inputs containing multiple intents. If the first intent needs a roll, the system should request that roll, narrate and persist its outcome, then continue into the next queued intent, which may itself require another roll.

The motivating failure is turn 5 of the 2026-05-10 Forty Thousand playthrough: the player gave two actions in one input. The system treated the input as one blended action, even though the first part could have required a social roll before the second part became valid. SF2 currently resolves a single deterministic action per turn and the Narrator route assumes one mutually exclusive `request_roll` or `narrate_turn` outcome per model call. Roll resumption can call the Narrator again, but there is no explicit ordered intent queue.

## Definitions

- **Intent queue** - ordered list of player intents parsed from one input, each with text, action type, target refs, and dependency on earlier intents.
- **Current intent** - the intent being resolved right now.
- **Remaining intents** - intents preserved across a roll pause or narrate continuation, surfaced privately so the Narrator can continue after the current outcome.
- **Intent boundary** - a connector such as "then", "after that", "once X", "and then", bullet/numbered commands, or two clearly independent imperative clauses.
- **Multi-roll turn** - a single player input that legitimately produces two or more roll prompts because resolving the ordered intents crosses multiple uncertainty points.

## Acceptance Criteria

- [ ] Player input is parsed into one or more ordered intents before Narrator call.
- [ ] Existing single-intent behavior remains unchanged for normal inputs.
- [ ] Each intent gets deterministic action resolution, target refs, forbidden substitutions, and optional required-roll gate.
- [ ] When the current intent requires a roll, the roll prompt records remaining intents in resume context.
- [ ] After a roll resolves, Narrator receives the result plus the remaining intent queue and may request another roll before final narration.
- [ ] A single player input can produce 2+ roll prompts when two ordered intents each push against meaningful uncertainty.
- [ ] Narration makes the sequence clear: outcome of intent 1 first, transition into intent 2 second.
- [ ] State mutations from each intent are applied in order and do not overwrite each other.
- [ ] Suggested actions at the end reflect the post-queue state, not the first intent's pre-resolution state.
- [ ] A fixture covers: "pressure Sable, then call Maret" where the social pressure resolves before the call.
- [ ] A second fixture covers a non-roll first intent followed by a roll-gated second intent.
- [ ] A third fixture covers two roll-gated intents in one input.

## Historical Agent Brief (superseded by 2026-05-16 verification)

**Category:** enhancement
**Summary:** Support ordered multi-intent player inputs, including multiple roll pauses in one submitted turn.

**Current behavior:** SF2 resolves the player input as one blended action. A roll pause can resume the same action, but there is no explicit queue for "do X, then Y" inputs where later intents depend on earlier outcomes.

**Desired behavior:** Player input should be parsed into an ordered queue. The current intent resolves first, any required roll pauses with remaining intents stored in resume context, and later intents continue in order after the prior outcome is known.

**Key interfaces:** player-input intent parsing, resolved action metadata, required-roll gate data, roll prompt/resume context, ordered turn commit/state mutation flow, suggested-action generation, replay fixture assertions.

**Acceptance criteria:**
- [ ] Existing single-intent inputs behave as they do today.
- [ ] Multi-intent inputs preserve order, dependencies, and target refs across roll pauses.
- [ ] One input can legitimately produce multiple roll prompts before final narration.
- [ ] State mutations commit in sequence and final suggested actions reflect the post-queue state.
- [ ] Fixtures cover social-then-call, non-roll-then-roll, and two-roll sequences.

**Out of scope:** General natural-language planning AI, arbitrary background task scheduling, or changing the visible dice UI beyond what ordered roll pauses require.
