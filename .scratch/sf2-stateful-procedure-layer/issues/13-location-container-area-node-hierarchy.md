## What to build

Make SF2 locations hierarchical wherever exploration topology is load-bearing. A canonical location/topology is the parent container, and areas/nodes are children under that container. The system should represent:

```text
Shallowden Station
|-- Berth 14
|-- Cantina
|-- Customs Gate
```

instead of flattening the current place into a single location name such as `Berth 14, Shallowden Station`.

This should apply across genres: a space station with berths and concourses, a grimdark castle with gatehouse/bailey/crypt nodes, a cyberpunk network with auth gateway/admin subnet/data vault nodes, a fantasy ruin with chambers and sealed passages, or a noir district with precinct/bar/alley nodes.

## Definitions

- **Location/topology container** — the canonical parent entity: station, castle, district, ruin, ship, wilderness region, network, domain, settlement.
- **Area/node** — a child node inside a location/topology container: berth, cantina, gatehouse, crypt, subnet, vault, alley, bridge, shrine.
- **Current position** — should resolve to both a parent location/topology and an optional current area/node. UI may display either one line or two, but persistence should keep them separate.
- **Flattened place alias** — a prose or legacy label that combines child and parent, such as `Berth 14, Shallowden Station`. Validator/deduper should parse or alias it into parent + child when possible, not preserve it as a separate canonical location.

## Role and Dedup Contract

- **Archivist** may propose a new location/topology, area/node, or current position change, but should not encode parent and child as one canonical location when both are present.
- **Validator/deduper** resolves flattened place aliases into parent container plus child node when confidence is high. If confidence is low, it preserves the text as an alias and flags the ambiguity rather than merging unrelated places.
- **Retrieval** should include parent location/topology and current area/node separately when both exist.
- **UI** should render the hierarchy clearly enough that the player understands where they are without treating every room/berth/subnet as a top-level location.
- **Migration/normalization** should preserve existing flat location names but attach split aliases where the parent/child structure can be inferred safely.

## Acceptance criteria

- [ ] Location state can represent a parent location/topology with child area/nodes.
- [ ] Current position can reference both `locationId` and optional `areaNodeId` or equivalent scoped refs.
- [ ] Validator/deduper does not create a new top-level location for `Berth 14, Shallowden Station` when `Shallowden Station` exists and `Berth 14` is an area/node.
- [ ] Validator/deduper preserves parent/child identity scope: parent locations, child area/nodes, exits/edges, route constraints, hazards, and ambient alertness remain distinct records.
- [ ] Retrieval packets expose parent location/topology and current area/node separately.
- [ ] UI or diagnostics can render the hierarchy as parent with child areas/nodes rather than comma-concatenated flat locations.
- [ ] A replay fixture covers `Shallowden Station` with `Berth 14`, `Cantina`, and `Customs Gate` as child nodes.
- [ ] A genre-neutral fixture covers either a grimdark castle or cyberpunk network using the same parent/child shape.
- [ ] A migration/normalization fixture covers a legacy flat label and verifies it becomes an alias/link rather than a duplicate canonical location.
