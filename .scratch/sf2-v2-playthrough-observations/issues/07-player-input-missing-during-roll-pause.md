# Player input is hidden during roll-pause turn

Category: bug
Status: needs-triage

## Observation

On turn 2, the player's submitted input was not shown in the turn stream. The narration appeared immediately after the turn marker, and during the roll-pause state the narration visually read too much like the PC input styling.

## Expected

The player input for the active turn should remain visible as soon as it is submitted, including while the Narrator is paused on a roll request.

Narrator text should keep a distinct narrator treatment during live streaming and roll pauses.

## Lightweight Code Note

No fix attempted. Initial inspection points at the live-turn rendering path:

- `components/sf2/play-shell.tsx` renders player input only from committed history turns: `turn.playerInput && <PlayerMessage>...`.
- The live in-progress prose path renders only `prose` through `<GMMessage live>...`.
- `app/play/v2/page.tsx` pauses `runNarrator` when a `roll_prompt` arrives, before `commitSf2Turn` writes the turn to history.
- During that pause, the submitted `playerInput` has been cleared from the input box but is not yet present in history, so there is no rendered player bubble for the active turn.

## Triage Questions

- Should the active submitted input be rendered optimistically before commit, then reconciled with the committed turn?
- Should live Narrator prose have the exact same visual treatment as committed Narrator prose, or a subtler "streaming" variant?
- Should the roll request attach visually to the active narrator block or sit in the action area only?
