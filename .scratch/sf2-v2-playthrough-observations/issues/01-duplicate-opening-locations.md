# Duplicate opening locations in V2 UI

Category: bug
Status: needs-triage
Area: SF2 / V2 UI / location state
Reporter: Martin
Observed: 2026-05-02

## Observation

During a space opera SF2 playthrough, immediately after chapter start the V2 UI Locations rail showed three locations:

- Revik Station Bay 14
- Revik Station Bay 14
- Revik station cargo concourse

The two `Revik Station Bay 14` entries appeared to be literal duplicates, including the exact same subline description.

## Expected

The Locations rail should not show duplicate player-facing locations at chapter start. If two state records refer to the same place, the UI or state layer should surface one canonical entry.

## Lightweight Code Note

No fix attempted. Initial inspection suggests current dedup is ID-based rather than semantic:

- `components/sf2/play-shell.tsx` builds the Locations rail with a `Map` keyed by `location.id`, so different IDs with the same name/description still render as separate rows.
- `lib/sf2/validation/apply-patch.ts` has explicit create dedup for NPCs, but the `location` create branch writes a new location by ID without name/description matching.
- Chapter open seeds `loc_opening` from `openingSceneSpec` in `app/play/v2/page.tsx`; a later location write with another ID but the same display fields can coexist with it.
