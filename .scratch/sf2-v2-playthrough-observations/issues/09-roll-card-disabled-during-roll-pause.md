# Roll card cannot be tapped during roll pause

Category: bug
Status: needs-triage
Severity: blocker
Area: SF2 / V2 UI / roll flow
Reporter: Martin
Observed: 2026-05-02

## Observation

During a space opera SF2 playthrough, a roll request appeared with `Tap to roll`, but tapping it did nothing. This blocked the game from continuing.

## Expected

When a roll request is shown, the player must be able to tap the roll card and resolve the pending check. The game should then resume Narrator streaming with the roll result.

## Lightweight Code Note

No fix attempted. Initial inspection strongly suggests a UI deadlock:

- `app/play/v2/page.tsx` computes `busy = isStreaming || isArchiving`.
- `runNarrator` sets `isStreaming=true` at the start of the Narrator stream.
- When `roll_prompt` arrives, `runNarrator` sets `pendingCheck` and awaits a player roll, but does not set `isStreaming=false` during that pause.
- `components/sf2/play-shell.tsx` passes `busy` into `DiceTray`.
- `DiceTray.handleRoll()` returns early when `busy` is true.
- `RollCardView` receives `disabled={busy || isRolling}`.

So while the UI is paused awaiting the roll, `isStreaming` likely remains true, making `busy` true, which disables the roll card that is supposed to unblock the stream.

## Triage Questions

- Should pending roll resolution be allowed while `isStreaming` is true, since that state represents a paused stream awaiting input?
- Should the app introduce a distinct `isAwaitingRoll` / `isRollPaused` state instead of using `isStreaming` as both "model is writing" and "turn is active"?
- Is there an immediate safe workaround for active playthroughs, or does the current save need code intervention to continue?
